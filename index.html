Общая информация

Данная фронтенд-сборка предоставляет готовые решения и функционал для быстрого и удобного создания различных интерфейсов сайта. 90% (а то и больше) того, что как правило требуется при разработке сайта тут есть.

Сборка основана как на готовых библиотеках взятых на просторах интернета, так и собственноручно написанныx плагинах. Всё это собрано в кучу и разложено по полочкам так, чтобы всем этим было удобно пользоваться.

Решено колоссальное количество тонкостей и нюансов. Всё это основывается на многолетнем опыте, проб и ошибок.

Главной целью создания данной сборки являлось желание в разы ускорить разработку. А главное, чтобы все инструменты не были кастыльными и узконаправленными, а были по максимуму гибкими и удобными, чтобы была возможность создавать нетривиальне решения.

Абсолютно ко всему имеются подробные комментарии с пояснениями и советами.

Файловая структура стартовой сборки

По порядку пробежимся по всем файлам стартовой сборки. В корневой папке находятся следующие файлы:

Нижнее подчёркивание ( _ ) в названии файлов указывает на то, что это аля технический вспомогательный файл, который никак не участвуюет в отображении сайта. В них содержаться части кода, которые можно подсмотреть, скопировать и т.д. Не нужные файлы короче!

index.html

Данный файл содержит в себе весь HTML код текущей презентации. Все элементы, которые можно здесь наблюдать, реализованы через функционал стартовой сборки.

Все решения тех или иных элементов можно посмотреть здесь.

_start.html

Данный файл предназначен для того, чтобы не мучиться каждый раз удаляя весь лишний код демонстрационного шаблона из index.html при начале разработки нового сайта. Просто скопируйте в index.html код из _start.html и начинайте с чистого листа.

_form.html

Данный файл содержит в себе HTML код формы, демонстрирующий реализацию абсолютно всех функций форм стартовой сборки. Работает на функционале forms.js (подробно далее)

_modal.html

Данный файл содержит в себе HTML код модальных окон, которые работают на плагине modal.js (подробно далее). Здесь продемонстрированы все варианты объявления модальных окон всех типов.

Данный код как правило должен располагатья где-то вконце кода перед закрывающимся тегом body.

_pagging.html

Просто готовая HTML разметка пагинации на случай чего.

_items.html

Данный файл необходим, чтобы продемонстрировать некоторый функционал, о котором пойдёт речь в будущем.

Папка "scss"

Набор scss-файлов - это второй основополагающий столп данной сборки.

Многие разработчики к файам стилей относятся снисходительно и не уделяют им должного внимания. Но и их так же как и любой другой код необходимо "содержать в чистоте" и структурировать. Что здесь и выполнено в полной мере.

Так как данная сборка в отношении стилей основана на препроцессоре scss, то крути не крути, а чтобы по максимуму эффективно её использовать, необходимо scss знать и уметь им пользоваться.
В отличии от html файлов в корневой папке, нижнее подчёркивание в названии scss-файлов имеет конкретное значение и функционал.

Подробно о структуре SCSS

В данной структуре style.scss является главным файлом, через который происходит компиляция всех остальных файлов:

Как можете заметить, style.scss единственный без нижнего подчёркивания. Именно style.css будет скомпилирован в целевой дериктории. Если создать ещё какой нибудь .scss файл без нижнего подчёркивания, то он тоже будет скомпилирован с расширением .css
Все стили прописаны по методологии БЭМ
Для смыслового разделения стилей рекомендуется разделять их по отдельным файлам. Например если создаётся новый раздел "Каталог", то создайте новый файл _catalog.scss и подключите его в style.scss

Файл "style.scss"

Пробежимся по содержимому данного файла:

Константы

// Константы сетки сайта
$containerWidth: 800;
$containerPadding: 15;
$minWidth: 320;
$maxWidth: 1920;
$maxWidthContainer: $containerPadding * 2 + $containerWidth;
$md0: $maxWidthContainer + 12;
$md1: 1200;
$md2: 991;
$md3: 767;
$md4: 479;

// Основной шрифт сайта
$fontFamily: "Arial";

$containerWidth - Данная переменная отвечает за ширину основного контейнера сайта.

$containerPadding - Отступы контейнера, чтобы на малых разрешениях он не прилипал к краям.

Как начать? Синхронизация по FTP

Синхронизация по FTP

Если вы работаете с какими либо файлами по удалённому доступу, то чтобы каждый раз не мучиться с заливкой файлов обратно на сервер после каждого их изменения, желательно настроить синхронизацию локальных файлов с удалёнными, чтобы всё это происходило автоматически.

Для этого необходимо установить в VS Code следующий плагин:

  • SFTP

Настройка синхронизации

Для этого необходимо заполнить конфигурационный файл SFTP плагина (Ctrl + Shift + P > SFTP: Config):

{
  "name": "",
  "host": "",
  "protocol": "ftp",
  "username": "",
  "password": "",
  "remotePath": "/",
  "uploadOnSave": true,
  "downloadOnOpen": true,
  "watcher": {
    "files": "assets/**/*",
    "autoUpload": true,
    "autoDelete": false
  },
  "ignore": [".vscode", ".git", ".gitignore", "readme.md"]
}
Заполните значения "name", "host", "username", "password", "remotePath" в соответствии с вашим удалённым сервером. После сохранения синхронизация будет работать.
Обратите внимание на установку "watcher". Она необходима для корректной синхронизации файлов стилей. Без данной настройки автоматически скомпилированные файлы заливаться на сервер не будут. Так что будьте внимательны с путём, куда смотрит watcher.

Синхронизация по FTP Якоря

Якоря

Якоря обеспечивают плавный переход к указанному элементу по клику.

Плавный переход может не отрабатывать из-за настроек браузера.

Якорем может быть любой HTML элемент.

Чтобы создать якорь, необходимо элементу по которому будет производиться клик добавить следующий атрибут:

data-goto="класс элемента к которому будет осуществляться переход".

Без точки!

В данной презентации таким образом реализована иконка возврата в начало страницы:

<button class="move-up" type="button" data-move-up data-goto="header" data-fix-m></button>

Обратите внимание на атрибут data-move-up. По данному атрибуту изменяется состояние иконки, после того, как страница проскролена больше чем на высоту экрана. В зависимости от этого иконке добавляется класс, по которому она либо скрывается, либо показывается.

Также якорю дополнительно можно задать атрибут data-offset="целочисленное значение". Данный параметр определяет отступ от верхнего края скрина при переходе к элементу.

Это одиночный якорь. Также есть возможность создать блок якорей.

Блок якорей

Чтобы это сделать, необходимо родительскому блоку в котором имеется несколько якорей, добавить атрибут data-goto-group.

Смысл этого в том, чтобы при скроле по странице выделять тот якорь, элемент которого находится в зоне видимости. Якорю будет добавляться класс _highlight, через который в стилях это можно будет сделать.

В данной презентации таким образом реализовано навигационное меню:

<ul class="navigation-menu__list" data-goto-group>
  <li class="navigation-menu__item" data-menu-item>
    <a class="navigation-menu__link" data-goto="header__section_main-menu">Многоуровневое меню</a>
  </li>
  <li class="navigation-menu__item" data-menu-item>
    <a class="navigation-menu__link" data-goto="header__section_menu">Простое меню</a>
  </li>
  <li class="navigation-menu__item" data-menu-item>
    <a class="navigation-menu__link" data-goto="header__section_simple-menu">Меню</a>
  </li>
  ...
</ul>

Подгрузка контента при скролле

В данной сборке реализована возможность ассинхронной подгрузки контента с сервера при скролле, или просто по нажатию на кнопку.

Пример реализации можно посмотреть в блоке "Подгрузка контента по нажатию на кнопку, плюс анимация" и "Подгрузка контента по скроллу, плюс анимация".

1. Подгрузка по нажатию на кнопку

Чтобы реализовать данный механизм, необходимо сделать следующее:

Элементу, по нажатию на который будет производится подгрузка контента, необходимо установить атрибут
data-content-loader="класс контейнера, в который будет производиться подгрузка".

Без точки!

Контейнеру, в который будет подгружаться контент необходимо установить атрибут
data-dynamic-content="строка запроса на сервер".

Дополнительно к строке запроса будет добавлен параметр itemsLoaded={количество элементов в контейнере}.

После обработки запроса, сервер должен вернуть HTML с контентом, который будет добавлен в конец блока.

<div class="loaded-content">
  <div class="loaded-content__body" data-dynamic-content="./_items.html">
    ...
  </div>
  <div class="loaded-content__bottom">
    <button class="btn" type="button" data-content-loader="loaded-content__body">Подгрузить контент</button>
  </div>
</div>
Для демонстрации данного функционала используется файл "_items.html".

2. Подгрузка по скроллу

Чтобы реализовать данный механизм, необходимо сделать следующее:

Контейнеру, в который будет подгружаться контент необходимо установить атрибут
data-scroll-dynamic="строка запроса на сервер".

Дополнительно к строке запроса будет добавлен параметр itemsLoaded={количество элементов в контейнере}.

После обработки запроса, сервер должен вернуть HTML с контентом, который будет добавлен в конец блока.

Всё. Теперь у нас реализован бесконечный скролл.

<div class="loaded-content">
  <div class="loaded-content__body" data-scroll-dynamic="./_items.html" data-offset="100" data-load-disable>
    ...
  </div>
</div>

Есть возможность создать элемент, при нажатии на который будет переключаться возможность подгрузки контента по скроллу.

<button class="btn" type="button" data-dynamic-loader></button>

Для этого такому элементу необходимо добавить атрибут data-dynamic-loader (вне контейнера).

Как правило сразу после контейнера.

Дополнительные атрибуты

Также есть возможность установить контейнеру следующие атрибуты:

data-offset="смещение нижней границы контейнера до нижнего края экрана при котором происходит подгрузка контента".

data-load-disable - запретить подгрузку

Медиа-фреймы Меню

Меню

Для организации меню любой сложности, в данной сборке используется собственный плагин menu.js.

Небольшая вводна информация

На создание меню, а в особенности сложных, многоуровневых меню, при разработке сайта уходит львиная доля времени. Причем в данной задаче очень много подводных камней. А что если на странице меню несколько? А что если они все должны себя вести по разному? А как они должны функционировать на различных устройствах при разных разрешениях экрана? Нюансов очень много.

Поэтому, при разработке данного плагина целью было - обеспечить простое, но эффективное решение всех этих вопросов. А главное чтобы это никак не влияло на гибкость по созданю меню. Ведь меню это настолько индивидуальная сущьность, что под неё очень трудно создать что-то универсальное. Поэтому пришлось идти на некоторые компромисы.

Если вы ожидаете, что сейчас накидаете HTML разметку, объявите экземпляр класса и, вуаля, у вас уже красивое меню... это не так.

Компромис заключается в том, что плагин не использует совершенно никаких стилей. Он вообще ни коим образом не влияет на визуал. Всё визуальное представление создаваемого вами меню целиком и полностью ложится на вас, и зависит от того, как хорошо вы обращаетесь со стилями.

Задача данного плаина заключается лишь в том, чтобы корректным образом, в определённых ситуациях, на определённые события, тем или иным элементам добавлять или удалять необходимые классы, по которым вы уже в стилях будете работать и определять визуал.

И как бы просто это не звучало, под капотом плагина достаточно объёмный механизм.

Структура data-атрибутов

  1. data-menu="любое уникальное имя"

    Основной атрибут по которому будет объявляьтся экземпляр класса Menu.

  2. data-menu-icon="тоже самое имя, что и у data-menu"

    Атрибут устанавливается на элемент, по нажатию на который будет раскрываться и сворачиваться меню.

    В данном контексте это означает, что будет добавляться/удаляться класс, по которому в стилях необходимо будет определить поведение.

    Данный элемент находится за пределами основного контейнера меню (data-menu) в HTML разметке.

    Как видите, через имя меню в data-атрибутах происходит связывание элементов. Тем самым происходит разделение функционала в том случае, если меню со своими связанными элементами на странице несколько.

  3. data-menu-container="тоже самое имя, что и у data-menu"

    Элемент, на который установлен данный атрибут, будет являться контейнером меню, который будет скрываться при определённых разрешениях экрана (мобильная версия), и разворачиваться по нажатию на data-menu-icon.

    В данном контексте это означает, что будет добавляться/удаляться класс, по которому в стилях необходимо будет определить поведение.
    Данный атрибут можно задать любому элементу в структуре DOM, не важно где он находится по отношению к data-menu. Он лишь определяет, у какого элемента будет добавляться или удаляться класс.
    Элементам data-menu-icon и data-menu-container будет добавляться/удаляться класс "_active"

  4. data-menu-item

    Данный атрибут устанавливается на пункты меню.

    Название меню атрибуту прописывать не нужно, так как он всегда находится внутри экземпляра Menu.
    На устройствах с тачскрином, при активном пункте добавляетя класс "_active"

  5. data-menu-control

    Данный атрибут устанавливается на элемент, который будет раскрывать подменю следующего уровня. Как правило это кнопка в виде стрелки.

    Название меню атрибуту прописывать не нужно, так как он всегда находится внутри экземпляра Menu.
    При наведении курсора мыши на пункт меню, или при клике на устройствах с тачскрином, добавляется класс "_hover"

  6. data-menu-submenu

    Данный атрибут устанавливается на элемент, являющийся подменю.

    Название меню атрибуту прописывать не нужно, так как он всегда находится внутри экземпляра Menu.
    При наведении курсора мыши на пункт меню, или при клике на data-menu-control на устройствах с тачскрином, добавляется класс "_visible"

  7. data-menu-close

    Данный атрибут устанавливается на элемент внутри data-menu-container, по нажатию на который меню закрываться.

    В данном контексте это означает, что будет удаляться класс, по которому в стилях необходимо будет определить поведение.
    Название меню атрибуту прописывать не нужно, так как он всегда находится внутри экземпляра Menu.

  8. data-menu-back (опционально)

    Устанавливается на элемент, при нажатии на который меню будет возвращяться к предыдущему уровню.

    В данном контексте это означает, что будет удаляться класс, по которому в стилях необходимо будет определить поведение.
    Название меню атрибуту прописывать не нужно, так как он всегда находится внутри экземпляра Menu.

Дополнительные классы

При загрузке страницы всем подменюшкам дополнительно будет добавлен один из классов "_toleft" или "_toright". В зависимости от того, в какую сторону разворачивать подменюшку если она уходит за пределы экрана. Приоритет за "_toright".

Объявление экземпляра класса меню

const menu = new Menu('идентификатор меню указанный в data-menu', {
  lockOnOpen: true,
  closeOnSelect: false,
  closeOnOutside: true,
  isOpen: false,
});

  1. lockOnOpen

    Запрет скроллинга по странице при открытом меню. По умолчанию: true.

    В данном контексте это означает, будет ли добавлен тегу body соответствующий класс.
    При открытии меню тегу body будет добавлять класс '_lock', а при закрытии удаляться.

  2. closeOnSelect

    Будет ли меню закрываться при клике на один из его пунктов. По умолчанию: false.

    В данном контексте это означает, что будет добавляться/удаляться класс, по которому в стилях необходимо будет определить поведение.

  3. closeOnOutside

    Будет ли меню закрываться при клике вне его области. По умолчанию: true.

    В данном контексте это означает, что будет добавляться/удаляться класс, по которому в стилях необходимо будет определить поведение.

  4. isOpen

    Будет ли меню открыто после загрузки страницы. По умолчанию: false.

    В данном контексте это означает, что будет добавляться/удаляться класс, по которому в стилях необходимо будет определить поведение.

Создание спойлеров

Функционал плагина опирается на data-атрибуты, которыми определённым образом необходимо разметить спойлер.

<div class="spoiler" data-spoiler="spoiler">
  <div class="spoiler__item" data-spoiler-item>
    <button class="spoiler__control" type="button" data-spoiler-control>Title #1</button>
    <div class="spoiler__container" data-spoiler-container>
      <!-- Любой контент -->
    </div>
  </div>
  <div class="spoiler__item" data-spoiler-item data-spoiler-open>
    <button class="spoiler__control" type="button" data-spoiler-control>Title #2</button>
    <div class="spoiler__container" data-spoiler-container>
      <!-- Любой контент -->
    </div>
  </div>
  ...
</div>

За счёт того, что функционал завязан на data-атрибуты, достигается исключительная гибкость в построении спойлеров. Вы никак не привязаны к какой бы то ни было жесткой структуре HTML разметки. И предоставленный пример кода показывает лишь общую структуру. Вам ничего не мешает влепить любой код куда угодно внутри data-spoiler. Единственное условие, это чтобы внутри data-spoiler-item был один data-spoiler-control и один data-spoiler-container

Data-атрибуты спойлера

  1. data-spoiler="уникальное имя спойлера"

    Основной атрибут по которому будет объявляьтся экземпляр класса Spoiler.

  2. data-spoiler-item

    Элемент спойлера, в котором содержится раскрывающийся контент и управляющий элемент, по нажатию на который контент будет сворачиваться или раскрываться.

  3. data-spoiler-control

    Управляющий элемент спойлера, по нажатию на который будет расскрываться соответсвующий контент.

  4. data-spoiler-container

    Разворачивающийся контейнер.

    Весь контент внутри data-spoiler-container будет обёрнут в дополнительный блок. Это необходимо для адекватной анимации сворачивания/разворачивания.

  5. data-spoiler-open (опционально)

    Данный атрибут можно задать элементу спойлера (data-spoiler-item). Тогда начальное состояние его будет открытым.

Объявление экземпляра класса спойлера

const spoiler = new Spoiler('идентификатор спойлера указанный в data-spoiler', {
  speed: 300,
  single: false,
  open: false,
  mediaMin: null,
  mediaMax: null,
});

  1. speed

    Скорость сворачивания/разворачивания спойлера. По умолчанию: 300 (милисекунд)

  2. single

    Позволяет быть развёрнутому только одному элементу. По умолчанию: false

  3. open

    Будут ли элементы спойера открытыми после загрузки страницы. По умолчанию: false

  4. mediaMin

    Минимальное разрешение экрана, на котором спойлер будет работать. В противном случае он будет отображаться как обычный блок. По умолчанию: null

  5. mediaMax

    Максимальное разрешение экрана, на котором спойлер будет работать. В противном случае он будет отображаться как обычный блок. По умолчанию: null

Примеры реализации можно посмотреть в блоках "Спойлер", "Аккордеон", "Многоуровневое меню на основе спойлеров".

За счёт того, что функционал завязан на data-атрибуты, достигается большая гибкость в построении табов. Внутри data-tabs может быть всё что угодно. Необходимо соблюдать всего одно условие - Количество элементов в data-tabs-controls должно сответствовать количеству элементов в data-tabs-container.

Data-атрибуты табов

  1. data-tabs="уникальное имя табов"

    Основной атрибут по которому будет объявляьтся экземпляр класса Tabs.

  2. data-tabs-controls

    Контейнер, содержащий в себе n-ое количество закладок.

    Закладки должны определятья через тег button. Это сделано для того, чтобы была возможность переключаться между закладками посредствам клавиатуры.

  3. data-tabs-container

    Контейнер, содержащий в себе n-ое количество панелей с контентом, открывающимися по нажатию на соответствующую закладку.

    Внутри data-tabs может быть любое количество data-tabs-container. Главное, чтобы количество панелей в них соответствовало количеству закладок в data-tabs-controls.
    Множественные data-tabs-container не работают при включённой функции AJAX

  4. data-hash="название закладки" (опционально)

    Устанавливается на кнопки (закладки) которые находятся в data-tabs-controls.

    Позволяет управлять панелью закладок через хеш (см. методы класса).

  5. data-action="строка запроса" (опционально)

    Устанавливается на кнопки (закладки) которые находятся в data-tabs-controls для ассинхронной подгрузки контента в контейнер.

    Используется при AJAX режиме

Объявление экземпляра класса табов

const tabs = new Tabs('идентификатор табов указанный в data-tabs', {
  blinkSpeed: 300,
  startTabIndex: 0,
  ajax: false,
  test: false,
});

  1. blinkSpeed

    скорость анимации мерцания при переключении закладок. По умолчанию: 300 (милисекунд)

  2. startTabIndex

    Начальная открытая закладка. По умолчанию: 0

  3. ajax

    При включеном режиме AJAX закладки будут работать через ассинхронную подгрузку контента с сервера. По умолчанию: false

    При использовании данного режима, контейнер data-tabs-container необходимо оставить пустым.
    <div class="tabs" data-tabs="tab">
  <div class="tabs__nav" data-tabs-controls>
    <button class="tabs__btn" type="button" data-action="строка запроса 1">Закладка один</button>
    <button class="tabs__btn" type="button" data-action="строка запроса 2">Закладка два</button>
    <button class="tabs__btn" type="button" data-action="строка запроса 3">Закладка три</button>
    ...
  </div>
  <div class="tabs__content" data-tabs-container></div>
</div>

  4. test

    Технический параметр. При включенном режиме AJAX, если на сервере ещё нет обработчика запросов в табах, то в консоли будет выскакивать ошибка. Если true, то запрос на сервер отправляться не будет.

Методы класса

  1. setTabByIndex(value)

    Установить закладку по порядковому номеру (отсчёт с нуля)

  2. setTabByHash(value)

    Установить закладку по порядковому номеру

    Закладки должны быть размечены атрибутом data-hash="значение"

Примеры реализации можно посмотреть в блоках "Табы", "Вложенные табы", "AJAX табы".

Если какому-то елементу внутри модального окна установить атрибут data-close, то по его нажатию модальное окно будет закрываться.

Элементу, по нажатию на который модальное окно будет открываться необходимо задать атрибут data-modal-opener="имя модального окна".

Тип тега не важен

Есть один стилистический нюанс. Когда модальное окно открывается, происходит фиксация скролла по странице. Из-за этого вертикальный скроллбар пропадает и фиксированные элементы на странице дёргаются на ширину этого скроллбара. Чтобы этого не происходило таким элементам необходимо задать один из атрибутов: data-fix или data-fix-m в зависимости от того, добавлять ли элементу badding или margin.

Таким элементом, допустим, является иконка возврата в начало страницы:

<button class="move-up" type="button" data-move-up data-goto="header" data-fix-m></button>

Табы Формы

Формы

Функционал форм, это пожалуй самая объёмна часть в сборке. Здесь решено колоссальное количество вопросов, нюансов, проблем, и всё это сведено в очень простую и удобную для использования форму.

Для организации форм используется файл forms.js. Это не плагин и не библиотека. Это набор скриптов, которые однако используют львиную долю всех библиотек и плагинов в этой сборке.

За визуал элементов формы отвечает файл стилей _forms.scss. Все стили структурированы, так что в большинстве случаев от вас потребуется только переопределить некоторые из них под стили дизайна.

Для того чтобы весь функционал скриптов начал работать, форму необходимо проинициализировать добавив ей атрибут data-form

<form class="form" action="#" method="post" data-form>
  ...
</form>

Инициализация по data-атрибуту реализована затем, чтобы если по какой-то причине весь дополнительный функционал форме не нужен, была возможность этого не добавлять.

Помимо прочих обязательных атрибутов, элемену формы можно также задать следующие атрибуты:

  1. data-send="test или ajax"

    test - тестовая отправка формы. Данные никуда не отправляются.

    ajax - ассинхронная отправка формы в соответствии с установками.

    В обоих случаях после сабмита открывается результатирующее модальное окно с именем form-sended. См. "Модальные окна".

  2. data-before="название функции" - функция, которая будет выполнена перед отправкой формы.

    Название функции без круглых скобок "()"

    Если функция вернёт false, форма отправлена не будет.

  3. data-after="название функции" - функция, которая будет выполнена после получения ответа с сервера.

    Название функции без круглых скобок "()"

    В параметре функции будут доступны данные, которые вернул сервер.

  4. data-validation - если атрибут установлен, то перед отправкой форма будет валидироваться.

  5. data-autocomplete-off - удаляет функцию автозаполнения полей формы браузерами.

Формы Валидация форм

Валидация форм

Механизм валидирования форм, это пожалуй самая мощьная разработка в этой сборке, которая основана на библиотеке justvalidate.min.js

Документацию можно посмотреть здесь: https://just-validate.dev/docs/intro.

Примеры можно посмотреть здесь: https://just-validate.dev/examples.

Каждый input формы будет валидироваться по разному в зависимости от установленного в нём значения атрибута type (text, email, password, checkbox, radio, file). См. документацию just-validate.
Все data-атрибуты указанные далее можно комбинировать между собой.
Если поле не прошло валидацию, к нему будет создан блок по отображению текстового сообщения об ошибке. Для адекватного отображения ошибки следите за вложенностью HTML разметки.

  2. data-min="значение"

  3. data-max="значение"

    Максимальное количество символов в поле.

Переключатели

Потенциал механизма переключателей формы разработанный в данной сборке трудно переоценить.

Бывают случаи, когда в форме есть чекбоксы или радиокнопки, а также селекторы, по нажатию на которые показываются или скрываются дополнительные поля. Или даже целые блоки дополнительных полей формы. И данный механизм переключателей осуществляет очень простую, и очерь гибкую работу с такими элементами. Благодаря переключателям, формой можно вертеть как захочется.

Особенностью переключателей является то, что они управляют не только отображением блоков с полями формы, но так же и управляют их валидацией. Блок скрыт - валидации нет. Блок показан - валидация есть.

И ещё одна особенность в том, что все это завязано на кастомные элементы. Кастомные радиокнопки, чекбоксы и селекторы.

Примеры реализации можете посмотреть в блоке "Форма".

Как всем этим пользоваться

Радиокнопке или чекбоксу, а также опции селектора, нужно задать атрибут data-switcher="уникальный идентификатор".

А блоку, в котором находится поле, или несколько полей формы, задать атрибут
data-switch="такой-же идентификатор, как и в data-switcher", или
data-switch-rev="такой-же идентификатор, как и в data-switcher".

Допустим любой уровень вложенности тегов.
К одному data-switcher может быть привязано сколько угодно data-switch и data-switch-rev.

В этом случае, если указанный data-switcher активен, то сответсвующие ему data-switch будут отображаться и валидироваться, а data-switch-rev наоборот скрываться и невалидироваться.

И соответственно наоборот. Если указанный data-switcher не активен, то сответсвующие ему data-switch будут скрываться и не валидироваться, а dta-switch-rev наоборот показываться и валидироваться.

В блоках data-switch и data-switch-rev должен находитья по крайней мере один любой элемент формы. Иначе работать не будет. Если нужно сделать обычный скрывающийся/показывающийся блок без элементов форм, то просто создайте в нем инпут с типом hidden.
Блоки скрываются по средствам задания стилей по селектору [data-switch][data-disable] и [data-switch-rev][data-disable] соответственно в _forms.scss.

Изначально в index.html скрипт google reCAPTCHA уже подключен. Вам только необходимо вставить туда соответствующий ключ.

Чтобы установить рекапчу, в форме необходимо создать два скрытых инпута cледующего формата:

<input type="hidden" name="siteKey" value="ключ сайта, полученный при регистрации домена на https://developers.google.com/recaptcha/docs/v3">
<input type="hidden" name="token" value="">

После инициализации, значение поля input[name="token"] будет заполнено неким хэшом, который необходимо будет обработать на стороне сервера после отправки формы, используя второй ключ secretKey, полученный при регистрации домена на https://developers.google.com/recaptcha/docs/v3.

Как правильно обрабатывать рекапчу на стороне сервера так же смотрите документацию.

Уникальные радиокнопки

Так же в сборке разработан механизм, который позволяет придавать абсолютно уникальный вид различным радиокнопкам.

Пример исполнения можно посмотреть в блоке "Форма" в пункте "Полностью кастомные радиокнопки (чекбоксы)"

Для того, чтобы этого добиться необходимо сделать следующее:

Создать инпут с типом radio и атрибутом data-radio, а также дополнительно прописать ему атрибут data-custom:

<input class="custom-radio-class" type="radio" name="inputName" value="value" data-radio data-custom>

Сразу же после инпута создать блок span с атрибутом data-radio-custom в котором уже прописывать любую HTML разметку:

<span class="custom-radio-class__content" data-radio-custom>
  <!-- любой HTML код -->
</span>

И далее верстать его так, будто input является его родителем, используя любые классы для дальнейшей стилизации.

Пользовательский класс, который был указан в инпуте передастся родительскому лейблу, а сам input получит тот же самый класс с расширением класс__input
<label class="custom-radio-class radio radio_custom">
  <input class="custom-radio-class__input" type="radio" name="inputName" value="value" data-radio data-custom>
  <span class="custom-radio-class__content" data-radio-custom>
    <!-- любой HTML код с использованием span и img -->
  </span>
</label>
Конечно, любую HTML разметку, это громко сказано. Есть важный нюанс!

Так как input и span будут обёрнуты в label, то имеются семантические нюансы в HTML разметке. В теге label могут находиться только такие теги как input, span, img, а также другие менее часто используемые тэги. Поэтому всю разметку необходимо выполнять исключительно ими. А уже затем, в стилях, по классам можно любому спану задать то отображение, которое необходимо: block, flex, grid и т.д.

Активный элемент будет получать класс _checked, с которым и нужно работать в дальнейшем при стилизации.

Динамическая адаптация DOM

Для реализации динамической адаптации DOM в данной сборке используется собственный плагин adapt.js.

Инициализация

В сборке уже инициализировано.
new Adapt();

Использование

С помощью данного функционала можно перекидывать HTML блоки в другие блоки в зависимости от разрешения экрана.

В особенности это очень помогает в адаптации сложных заголовков сайта, с множеством сущностей. Теперь расположить те или иные блоки по порядку и красиво, а главное без размножения лишних сущьностей, становится очень просто.

Всё что нужно сделать, это блоку, который необходимо куда-то перебросить задать атрибут
data-adapt="куда, когда, каким по порядку".

Первым параметром задаётся класс блока, куда будет переносится текущий блок.

Без точки!

Вторым параметром задаётся разрешение экрана, меньше которого блок будет перенесён. Если не задавать то по умолчанию 767.

Третим параметром задаётся порядковый номер блока в целевом блоке. По умолчанию last. Можно задать first, или числовой индекс (отсчёт с единицы).

Пример реализации смотрите в блоке "Демонстрация динамического изменения DOM"
